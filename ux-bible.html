Section 01

Product Foundation

Vision, principles, architecture, AI model, and data systems that govern the entire product.

1.1 Product Vision

Household Food OS is an AI-powered household food operating system. It is not a recipe app, a meal planner, a grocery list manager, or a food waste tracker. It is an ambient intelligence layer that continuously maintains a living model of every household's food reality and surfaces the most valuable decision at every moment.

The system continuously answers five fundamental household questions:

Question 1
What should I cook? — Based on what's available, what's expiring, and what the household needs nutritionally.
Question 2
What will expire soon? — Continuous monitoring of inventory age and projected consumption rates.
Question 3
How do I achieve my nutrition goals? — Meal optimization against protein, fiber, calorie, and health targets.
Question 4
What groceries should I buy? — Forward-looking purchase intelligence based on planned meals and consumption patterns.
Question 5
What surplus can I exchange locally? — Community matching of excess inventory with nearby demand.

1.2 Product Principles

Principle 1
Inventory First. Every recommendation, every plan, every alert begins with what the household actually has.
Principle 2
One Decision at a Time. Present the single highest-value action at any moment. Never overwhelm.
Principle 3
Circular Architecture. Every user action returns to inventory. Every loop closes.
Principle 4
Progressive Disclosure. Show the result first, the reasoning on demand, the full system only when requested.
Principle 5
Never Dead-End. Every error, empty state, and failed action must suggest a next step.
Principle 6
Community is Optional. Community features never block core functionality. They are additive, not mandatory.
Principle 7
AI is Infrastructure. AI is not a feature or a chat bot. It is the invisible backbone of every recommendation and every alert.
Principle 8
Trust Through Transparency. Every AI decision must be explainable. Users must always understand why they're seeing something.
Principle 9
Household as Unit. The system models a household, not an individual. Multiple members share one pantry brain.

1.8 AI Architecture

The AI is not a single model — it is a coordinated system of specialized reasoning engines that share a unified household memory model.

AI System Memory — Always Active
  • Inventory State — Real-time pantry with confidence scores per item
  • Consumption Patterns — What was eaten, when, by whom, and at what rate
  • Shopping Behavior — Purchase frequency, preferred platforms, brand tendencies
  • Preferred Meals — Cooked, liked, and saved recipes with rating history
  • Rejected Meals — Never-recommend list based on explicit rejection or low feedback scores
  • Nutrition Progress — Daily and weekly macro tracking against goals
  • Community Behavior — Surplus patterns, exchange history, trust score
AI Inputs (Shared Across All Engines)
  • Current pantry inventory with quantities and expiry dates
  • Household profile: size, age groups, cooking frequency, budget
  • Dietary constraints: type, allergies, restrictions, dislikes
  • Nutrition goals: protein target, calorie ceiling, health focus areas
  • Shopping history: purchase frequency, platform, spend patterns
  • Community availability: nearby listings and requests
  • Time of day, day of week, upcoming calendar events (if permitted)
AI Outputs (Specific to Engine)
  • Meal Recommendations — Best, Quick, Lazy, Protein, Budget variants
  • Shopping Suggestions — Item, quantity, timing, platform recommendation
  • Expiry Alerts — Item, urgency level, recommended action
  • Nutrition Guidance — Deficit identification, meal adjustments, macro coaching
  • Community Matches — Ranked surplus-to-need pairings with distance and fit score
  • Impact Metrics — Savings, waste prevented, meals recovered, nutrition success rate
  • Inventory Confidence Score — Staleness detection, refresh prompts

1.10 Household Memory Model

The Household Memory Model is the AI's long-term understanding of a specific household. It is constructed from behavioral signals across all modules and continuously updated.

// HOUSEHOLD MEMORY MODEL STRUCTURE { "inventoryState": { items: InventoryItem[], lastScanDate: ISO8601, confidenceScore: 0–100, // drops 5pts/day without update estimatedValue: currency }, "consumptionPatterns": { byItem: { itemName → { avgDailyConsumption, dayOfWeekVariance } }, byCategory: { category → consumptionRate }, wasteRateByItem: { itemName → percentWasted } }, "preferenceModel": { likedMeals: recipeId[], dislikedMeals: recipeId[], preferredCuisines: string[], cookingComplexityPreference: "simple" | "moderate" | "complex", lastFiveRatings: MealRecord[] }, "nutritionHistory": { dailyProtein: number[], // 30-day rolling dailyCalories: number[], weeklyGoalMissed: string[] // ["protein", "fiber"] }, "shoppingBehavior": { typicalShopDay: string, // "Saturday" avgSpendPerTrip: currency, overboughtItems: string[], preferredPlatform: string } }

1.11 Recommendation Engine Logic

The recommendation engine scores every available recipe against a weighted multi-objective function and surfaces the top result as the primary recommendation, with 3–4 alternatives in different optimization categories.

// RECOMMENDATION SCORING FORMULA RecipeScore = ( ExpiryScore × 0.35 // Uses highest-risk inventory items + NutritionScore × 0.25 // Closes daily nutrition gap + PreferenceScore × 0.25 // Matches household taste model + ConvenienceScore × 0.15 // Time, complexity, effort ) // RECOMMENDATION VARIANTS Best: MaxScore(combined formula above) Quick: MaxScore(ConvenienceScore × 0.7 + ExpiryScore × 0.3) where prepTime ≤ 20min Lazy: MaxScore(ConvenienceScore × 0.9) where steps ≤ 4 AND prepTime ≤ 10min Protein: MaxScore(NutritionScore × 0.8) where proteinContribution ≥ 30g Budget: MaxScore(CostScore × 0.7) where costPerServing ≤ thresholdFromProfile // HARD CONSTRAINTS (recipe excluded if ANY violated) EXCLUDE IF: recipe.ingredients CONTAINS allergens FROM DietaryProfile.allergies EXCLUDE IF: recipe.mealType CONFLICTS WITH DietaryProfile.dietType EXCLUDE IF: recipe.id IN preferenceModel.dislikedMeals EXCLUDE IF: recipe was cooked WITHIN last 72 hours (household.cookingFrequency check) EXCLUDE IF: inventory coverage for recipe.ingredients < 60% (missing too many items)

1.12 Nutrition Engine Logic

// NUTRITION GAP CALCULATION DailyGap = NutritionGoal.proteinTarget - Sum(MealRecord.nutritionActual.protein for today) FiberGap = NutritionGoal.fiberTarget - Sum(MealRecord.nutritionActual.fiber for today) CalorieRemaining = NutritionGoal.calorieTarget - Sum(MealRecord.nutritionActual.calories for today) // NUTRITION ALERT THRESHOLDS IF DailyGap.protein > 30g AND it is after 5pm → Surface protein-focused meal recommendation IF weeklyGoalMissedCount >= 3 → Show nutrition insight on Home IF CalorieRemaining < 200 at mealtime → Recommend light options only // WEEKLY NUTRITION SCORING (for Planner) WeeklyNutritionScore = Avg(DailyNutritionScore for 7 days) DailyNutritionScore = Σ(nutrientAchieved / nutrientTarget × weight) for [protein, fiber, calories]

1.13 Shopping Engine Logic

// PURCHASE FORECAST CALCULATION DaysUntilEmpty(item) = CurrentQuantity / AvgDailyConsumption(item) PurchaseUrgency(item) = IF DaysUntilEmpty ≤ 2 → "Must Buy" | ≤ 5 → "Should Buy" | ELSE → "Optional" // QUANTITY RECOMMENDATION RecommendedQty(item) = (AvgWeeklyConsumption × ShoppingFrequencyWeeks) × 1.1 // 10% buffer CAP AT: RecommendedQty ≤ MaxBeforeWaste(item) // from waste pattern model // OVERBUY PREVENTION IF item.wasteRate > 30% → Show warning "You waste X% of this monthly" IF item in pantry AND DaysUntilEmpty > ShoppingFrequencyDays → Remove from "Must Buy" // RESTOCK HEALTH SCORE RestockHealth = (ItemsAdequatelyStocked / TotalTrackedItems) × 100

1.14 Community Matching Logic

// COMMUNITY MATCH SCORING MatchScore(offer, household) = ( DietaryCompatibility × 0.40 // Does item fit household's diet? + ProximityScore × 0.30 // Closer = higher score + FreshnessScore × 0.20 // Days until expiry + TrustScore × 0.10 // Provider's trust rating ) // HARD CONSTRAINTS EXCLUDE IF: offer.items CONTAIN household.allergies EXCLUDE IF: offer.distance > household.maxCommunityRange (default: 5km) EXCLUDE IF: offer.expiryDate ≤ today (never show expired items) EXCLUDE IF: offer.status ≠ "available" // TRUST SCORE FORMULA TrustScore = (completedExchanges × 0.5) + (avgRating × 0.3) + (responseRate × 0.2)

Screen 3 Dietary Setup Module: Onboarding
Purpose
Define dietary constraints that become hard filters on all recipe recommendations, community listings, and planner meals. This is the most safety-critical onboarding screen — allergies must be handled with zero tolerance.
System Goal
Create the DietaryProfile entity. Initialize the constraint graph that governs all recipe filtering. Establish the allergen exclusion list that operates as a hard block.

Sections

Diet Type

Single-select card group: Vegetarian / Eggetarian / Non-Veg / Vegan / Jain. Each card has a representative icon. Default: none selected (user must choose). Affects which recipe database segments are available.

Health Restrictions

Multi-select chip group. Options: Diabetes / Pregnancy / Heart Health / Kidney Health / Gluten Free / Lactose Free. Each restriction maps to a recipe tag exclusion or inclusion rule. Tapping any restriction shows a brief tooltip explaining its impact.

Allergies

Tag input with autocomplete. Common allergens pre-listed: Peanuts / Soy / Shellfish / Tree Nuts / Milk / Eggs / Wheat / Sesame. User can add custom allergens. Allergen tags shown in red. This list operates as a HARD BLOCK — no recipe containing any allergen will ever appear in recommendations.

Disliked Foods

Search-and-add text field. User types a food name and adds it to a list. Soft filter — disliked foods reduce recipe score but do not hard-block unless the food is a primary ingredient (more than 40% of recipe composition).

AI Inputs
  • Diet type, health restrictions, allergens, disliked foods
AI Outputs
  • DietaryProfile entity. Constraint graph for recipe filtering. Allergen exclusion list committed to hard block layer.
⚠️ Critical Safety Note

Allergen data must be stored separately from preferences with a dedicated validation layer. Any recipe recommendation pipeline must query allergen block list BEFORE scoring begins — not during scoring. This prevents allergenic recipes from ever reaching the ranking step.

Screen 6 Initial Kitchen Scan Module: Onboarding
Purpose
Create the first inventory. This is the most important onboarding screen — without inventory, the Home Dashboard is empty and the product has no value proposition. Every method must be offered and made equally accessible.
User Goal

Fridge Photo. Open camera. User photographs open fridge. AI detects visible items. Best for fresh produce and dairy.
Method 2
Pantry Photo. Open camera. User photographs pantry shelves. AI detects packaged goods, staples, canned items.

Receipt Upload. Photo or file select. OCR extracts purchased items. Best for recent grocery runs.

AI Inputs
  • Natural language user query
  • Full pantry inventory state
  • Nutrition goals and today's progress
  • Weekly plan (if active)
  • Household dietary profile
  • Community listings (if query is community-related)
  • Shopping preferences (if query involves purchasing)
AI Outputs
  • Natural language response (personalized, specific)
  • Structured data cards embedded in response (recipe cards, inventory snippets, nutrition rings)
  • Suggested action buttons (contextual, limited to 3 max)
  • Deep links to relevant screens for exploration

AI Response Design Rules

  • Always specific, never generic: "You have 8 eggs" not "You have eggs."
  • Always actionable: Every response must end with at least one concrete next step.
  • Progressive disclosure: Key answer first, supporting detail on expand.
  • Confidence transparency: If AI is uncertain ("Your pantry data may be outdated — last scan was 4 days ago"), say so.
  • Length: Responses under 80 words unless the user asks for a recipe or detailed plan.

5.7 Responsive Behavior

The primary target is mobile (390px width). For tablet contexts (768px+), the following adaptations apply:

  • Home Dashboard: Two-column layout. Recommendation Hero occupies left half. Expiry + Shopping Signals occupy right half in a stacked list.
  • Planner: Week View expands to show more meal detail per cell without truncation. Day Detail becomes a side panel on tablet.
  • Community Map: Map takes 60% of width. Listing detail panel appears as a right-side panel at 40% width on tablet.
  • All list views: Two-column grid instead of single-column list on screens wider than 600px.
